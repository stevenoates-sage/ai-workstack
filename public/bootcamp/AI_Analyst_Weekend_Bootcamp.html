Pre-Work: Complete Before Saturday Morning

Complete the following steps before starting Saturday. Estimated time: 20–30 minutes. If you run into issues, the Common Issues table in Phase 1 covers the most likely fixes.

💡
Copying commands: Every command in a green code box has a copy button in the top-right corner — click it and the command is copied to your clipboard. Paste into your terminal with Ctrl+V (or right-click → Paste in PowerShell).
ℹ️
Terminal vs Claude Code — know the difference. This bootcamp uses two places to type commands. Learn to tell them apart:

▸ Terminal — Green code boxes are commands you run in the VS Code integrated terminal (Ctrl+` to open). These are system commands like git commit, winget install, and git status.

▸ Claude Code — Blue prompt boxes marked with 📝 are prompts you type into the Claude Code panel. Click the starburst icon (✳) in the Activity Bar to open it. These are natural language questions like "Analyze gtm_pipeline.csv."

If you type a terminal command into Claude Code (or vice versa), it won't work. When in doubt, check the box color: green = terminal, blue = Claude Code.

# Chart Style Skill
## Purpose
Define the visual standard for all charts this analyst produces.
## Rules
1. Chart titles must state the finding (e.g. 'North America Leads ARR at 58%'), not the topic ('ARR by Region').
2. Use dark background (#1A1A2E), green accent (#00D639), white text.
3. Always label data values directly on bar and column charts.
4. Sort bar charts from highest to lowest unless time-ordered.
5. Never use pie charts or 3D charts.
6. Use consistent font: Arial or sans-serif.
## Do Not
- Do not use default matplotlib or seaborn color palettes.
- Do not produce charts without titles.
- Do not use gridlines on bar charts.

# Data Quality Skill
## Purpose
Run data quality checks before every analysis and flag issues.
## Checks to Run
1. Null values: count missing values in each field. Flag any field with >5% nulls.
2. Date gaps: check for missing months or periods in time-series data.
3. Outliers: flag any ARR or deal value more than 3 standard deviations from the mean.
4. Duplicate records: check for duplicate IDs or identical rows.
5. Stage consistency: confirm stage_number matches stage name.
## Output
Produce a 3-line data quality summary before every analysis:
'Data quality: X records, Y fields. Issues: [list or None]. Proceeding.'

# Question Framing Skill
## Purpose
Decompose any business question into MECE hypotheses before analyzing.
## Process
1. Restate the question in one sentence.
2. Generate 3-5 mutually exclusive hypotheses that could explain the pattern.
3. For each hypothesis, state what data would confirm or refute it.
4. Rank hypotheses by likelihood given the available data.
5. Only then begin analysis, starting with the most likely hypothesis.
## Example
Question: 'Why did win rate fall in Q2?'
H1: Deal mix shifted to larger, harder deals (check avg deal size).
H2: New reps joined and dragged down overall rate (check tenure vs win rate).
H3: A specific vertical underperformed (check win rate by vertical).

# Metric Specifications Skill
## Purpose
Define GTM KPIs precisely so every calculation is consistent.
## Core Metrics
ARR (Annual Recurring Revenue): Sum of arr_value for Closed Won deals.
Win Rate: Closed Won count / (Closed Won + Closed Lost count) x 100.
SQO (Sales Qualified Opportunity): Opportunities at Stage 2 or above.
Pipeline Coverage: Total open pipeline ARR / Quarterly quota target.
Avg Deal Size: Total ARR / Count of Closed Won deals.
Avg Sales Cycle: Mean days from created_date to close_date (Closed Won only).
Quota Attainment %: Actual ARR / Quota target x 100.
## Formatting
ARR: always in $000s with one decimal (e.g. $48.0K).
Percentages: one decimal place (e.g. 34.2%).
Sales cycle: integer days (e.g. 67 days).

# Storytelling Skill
## Purpose
Structure every analysis narrative using the SCR framework.
## SCR Framework
SITUATION: What is the current state? (1-2 sentences of context.)
COMPLICATION: What has changed or is at risk? (The insight or finding.)
RESOLUTION: What should happen next? (A clear recommendation.)
## Rules

2. Chart titles are written as findings, not topics.

# Root Cause Agent
## Trigger: Run when a metric is off-target or anomalous.
## Steps
1. Identify the metric and period in question.
2. Decompose: drill into each dimension (region, vertical, rep, deal size).
3. For each dimension, calculate contribution to the total variance.
4. Identify the dimension with the highest contribution.
5. Drill one level deeper into that dimension.
6. State the root cause in one sentence: 'The [metric] declined because [cause].'
7. Produce a waterfall or breakdown chart showing contribution.

Git Quick Reference & Tutorial

Git is the version control system that tracks every change you make to your skill and agent files. This section covers the commands used in this bootcamp, plus how to back up your work to GitHub so you can share it or access it from another computer.

What is Git?

Think of git as an unlimited undo history for your entire project folder. Every time you run git commit, git takes a snapshot of every file. You can view any previous snapshot, compare what changed between two snapshots, or roll back to an earlier version if something breaks. Unlike saving a file (which overwrites the previous version), git keeps every version forever.

Key Concepts

ℹ️
Repository (repo) -- your project folder, once git is tracking it. Created with git init.

Commit -- a saved snapshot of all your files at a point in time. Each commit has a message describing what changed.

Stage -- marking files to be included in the next commit. git add -A stages everything.

Branch -- a parallel version of your project. The default branch is called main. You can create experiment branches without affecting your working version.

Remote -- a copy of your repo hosted online (e.g. on GitHub). Lets you back up your work and share it with others.

Commands Used in This Bootcamp

All git commands run in the VS Code terminal (Ctrl+` to open) -- never in Claude Code.

CommandWhat it doesWhen to use it
git initCreates a new git repo in the current folderOnce, at the start of Phase 1
git statusShows which files have changed since last commitBefore every commit -- check what will be saved

Backing Up to GitHub (Optional but Recommended)

GitHub is a free website that hosts git repositories online. Pushing your repo to GitHub means you have a backup, and you can share your skills and agents with colleagues by sending them the repo link. Here's how to set it up:

Step 1 -- Create a GitHub Account (if you don't have one)

  • 1
    Go to github.com and click Sign up.
  • 2
    Use your work email. Choose a username and password. Complete the verification.

Step 2 -- Create a New Repository on GitHub

  • 1
    Log in to GitHub. Click the + icon in the top-right corner and choose New repository.
  • 2
    Name it ai-analyst (or whatever you like).
  • 3
    Set it to Private (recommended -- keeps your work visible only to you and anyone you invite).
  • 4
    Do NOT check "Add a README" or ".gitignore" -- you already have these locally.
  • 5
    Click Create repository.

Step 3 -- Find Your Repository URL

  • 1
    After creating the repo, GitHub shows a Quick Setup page. You will see a URL that looks like this:
https://github.com/YOUR-USERNAME/ai-analyst.git
  • 2
    Copy this URL. You can also find it later by going to your repo page on GitHub and clicking the green Code button -- the URL is shown under the HTTPS tab.

Step 4 -- Link and Push from VS Code

Run the following commands in the VS Code terminal:

  • 1
    Link your local repo to GitHub (paste your actual URL):
git remote add origin https://github.com/YOUR-USERNAME/ai-analyst.git
  • 2
    Rename your branch to main. Older versions of Git name the default branch master, but GitHub expects main. This command renames it -- if it's already called main, it does nothing:
git branch -M main
  • 3
    Push all your commits to GitHub:
git push -u origin main
  • 4
    GitHub may ask you to sign in via a browser popup. Authorize VS Code and the push will complete.
  • 4
    Go to your GitHub repo page in a browser. You should see all your files -- CLAUDE.md, skills/, agents/, and the CSV data files.
💡
After the initial push: From now on, whenever you make a commit, just run git push to upload it to GitHub. No need to repeat the -u origin main part.
